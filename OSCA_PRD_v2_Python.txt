1.1 Problem Statement
OSCA at NAAPľVillamor currently manages attendance for student athletes and artists using paper log sheets, and equipment inventory using handwritten forms. This causes proxy attendance fraud, inaccurate records, delayed reporting, and no real-time equipment visibility.

1.2 Proposed Solution
A centralized, web-based platform with:
Ľ Biometric facial recognition (time-in / time-out dual-scan) via DeepFace or InsightFace
Ľ Barcode-based equipment borrowing & return with automated eligibility validation
Ľ Role-based access control (Admin, Coach, PE Instructor, Student)
Ľ Real-time dashboards, PDF/XLSX reporting, and audit logs



2. Tech Stack Decision: Python FastAPI vs NestJS

Why This Comparison Matters
Ľ Version 1.0 of this PRD recommended NestJS (Node.js / TypeScript) as the backend.
Ľ Version 2.0 switches to Python FastAPI. This section explains the trade-offs transparently so the team can make an informed decision.
Ľ Both are production-grade. The right choice depends on your team's strengths and the facial recognition requirement.
2.1 Head-to-Head Comparison

2.2 Summary Verdict
Recommendation: Python FastAPI ? Ś For This Thesis
Ľ FACIAL RECOGNITION IS THE CORE FEATURE. Python's DeepFace/InsightFace is significantly more accurate and easier to implement than face-api.js. This alone justifies the switch.
Ľ TEAM SKILLS. Computer Science and Information Systems students in the Philippines predominantly learn Python first. FastAPI is faster to implement correctly.
Ľ SCALE. OSCA is a small office with <200 concurrent users maximum. Python's GIL limitation is irrelevant at this scale.
Ľ TRADEOFF ACCEPTED. You sacrifice TypeScript end-to-end type safety and marginally lower raw throughput Ś neither matters for a capstone-scale institutional system.
Ľ NestJS WOULD BE BETTER if: (a) the team has strong TypeScript skills, (b) the facial recognition requirement did not exist, or (c) the system needed to scale to thousands of concurrent users.


3. Docker Ś Why It Is Essential for This Project

3.1 What Docker Does
Docker packages your application and ALL its dependencies (Python version, libraries, OS-level packages like OpenCV, PostgreSQL, Redis) into lightweight, portable containers. Every container behaves identically regardless of the host machine.

3.2 Docker Advantages Explained
#AdvantageWhat It Means for Your Project1Eliminates 'Works on My Machine'Your InsightFace model loads the same on your laptop, your classmate's PC, and the NAAP school server Ś no version conflicts2One-Command Deploymentdocker compose up starts everything: FastAPI, PostgreSQL, Redis, MinIO, Nginx Ś the entire system in seconds3Isolated ServicesPostgreSQL runs in its own container; if it crashes, your FastAPI app keeps running and vice versa4Environment ParityDevelopment, testing, and production use identical environments Ś no 'it broke in production but not locally'5Easy RollbackIf an update breaks something, roll back to the previous Docker image with one command Ś no manual reinstall6Reproducible BuildsNew developer joins the team? They run docker compose up and have a full working environment in minutes, not days7OpenCV & dlib Complexity SolvedInstalling OpenCV and dlib (for facial recognition) manually on Windows/Mac is notoriously painful. Docker's Linux container handles all native dependencies cleanly8Resource IsolationEach container gets defined CPU and RAM limits Ś prevents one service from starving others9PortabilityDeploy on school server today; move to AWS/DigitalOcean tomorrow Ś same Docker Compose file works10Professional StandardDocker is the industry standard for 2026 deployments. Including it in your capstone demonstrates professional-grade engineering practice

5.1 User Roles & Access Control

5.2 Attendance Module
Student Registration & Facial Enrollment
Ľ Profile: name, student ID, course, year level, medical info, emergency contacts
Ľ Facial enrollment: capture 5+ images at varied angles via react-webcam in browser
Ľ Backend processes images with InsightFace/DeepFace ? generates 512-dim embedding
Ľ Embeddings stored in PostgreSQL with pgvector; raw images stored in MinIO

Attendance Scanning (Kiosk)
Ľ Time-IN: student stands at kiosk webcam ? FastAPI compares live frame embedding to stored vector
Ľ Time-OUT: second scan required upon session completion
Ľ Liveness check (MiniFASNet) runs before recognition to prevent photo spoofing
Ľ Minimum cosine similarity threshold: 0.85 Ś configurable by Admin
Ľ Response must complete within 3 seconds; fallback alert to Admin after 3 failed attempts
Ľ Attendance record saved: student_id, session_id, time_in, time_out, confidence_score, activity_type


5.3 Inventory Module
Equipment & Borrowing ID Management
Ľ Admin registers equipment: name, category, description, condition, quantity
Ľ System generates Code-128 barcodes for equipment; QR Codes for Borrowing IDs
Ľ For non-labelable items: barcode assigned to container/storage unit
Ľ PE Instructors issued physical Borrowing ID card with QR Code Ś printed via Zebra GK420d

Borrow / Return Workflow
Ľ Step 1: PE Instructor scans Borrowing ID barcode at scanner station
Ľ Step 2: FastAPI validates eligibility (active account, no overdue items, within limits)
Ľ Step 3: Scan each equipment barcode to add to transaction
Ľ Step 4: Confirm Ś system logs instructor_id, items[], borrowed_at, expected_return
Ľ Return: Re-scan Borrowing ID ? scan returned items ? system marks Available
Ľ Overdue alerts: Celery scheduled task emails instructor + Admin after due date passes

8. Security Requirements

8.1 Biometric Data Protection
Ľ Face embeddings encrypted at rest using AES-256-GCM via SQLAlchemy column encryption
Ľ Raw facial images stored in MinIO with private bucket policies (no public access)
Ľ Comply with Philippine Data Privacy Act of 2012 (R.A. 10173) Ś biometric data classified as sensitive personal information
Ľ Data Retention Policy: raw images deleted after successful embedding generation (configurable 30-day window)
Ľ Audit log entry created for every face scan attempt (success or failure)

8.2 Application Security
Ľ HTTPS enforced; HTTP ? HTTPS redirect in Nginx
Ľ JWT access tokens (15-min expiry) + refresh tokens with rotation (7-day)
Ľ Argon2id password hashing via passlib
Ľ FastAPI rate limiting: 5 login attempts per 15 min, then 30-min lockout
Ľ SQL injection impossible Ś SQLAlchemy ORM uses parameterized queries exclusively
Ľ Input validation on all endpoints via Pydantic v2 models
Ľ CORS policy: only allows requests from the Next.js frontend domain
